以koa为例，讲讲各种网络安全问题

发表于 2018-09-01

网络安全问题的文章到处都是，不过对于前端来说，很多东西都停留在理论。毕竟很多内容的涉及到和后端交互。这篇博客假想有一个 Koa 后台，来结合实际梳理一下网络安全的各种东西。

XSS-跨站脚本攻击

XSS, 即为（Cross Site Scripting）, 中文名为跨站脚本, 是发生在目标用户的浏览器层面上的，当渲染DOM树的过程成发生了不在预期内执行的JS代码时，就发生了XSS攻击。
跨站脚本的重点不在‘跨站’上，而在于‘脚本’上。大多数XSS攻击的主要方式是嵌入一段远程或者第三方域上的JS代码。实际上是在目标网站的作用域下执行了这段js代码。
获取页面数据，获取Cookie（站酷），劫持前端逻辑（跳转页面）， 发送请求。甚至可以提交表单时注入到后台管理系统。
XSS分为反射型和存储型。

反射型

url参数直接注入的攻击方式。发出请求时，XSS代码出现在URL中，作为输入提交到服务器端，服务器端解析后响应，XSS随响应内容一起返回给浏览器，最后浏览器解析执行XSS代码。
举个例子：某网站登录时url是这样的：http://lizhangyu.com/id=李章鱼，网站后台（在koa中，获取参数非常简单:context.query.id，其中 context 为一个包含了请求对象和响应对象的对象，我们后面也会经常用到它）接收到参数id: 李章鱼后在浏览器渲染出你好，李章鱼。如果有人给我发一个url地址http://lizhangyu.com/id=<script>alert(1)</script>,这个时候我点击进去浏览器会将id参数渲染进页面，识别为一个脚本，实现XSS攻击。

存储型

存储到DB后读取时注入的攻击方式。看了上边的例子，你会发现反射型XSS攻击力并不强，毕竟如果有人诱使我点击一个奇怪网站很容易让人起疑心。而存储型XSS就厉害的多了。同样举个例子：某黑客在某网站发表评论<script>alert(1)</script>，这个评论存在服务器数据库里所有用户都能看到，如果网站没有任何处理措施，所有访问该网站的用户都会渲染出该脚本。可见，存储型XSS攻击危险性比反射型大很多。

有人会想，在页面中插入<script>alert(1)</script>能有多大危害？实际上，如果页面能够插入<script>alert(1)</script>,那就说明我们能在用户浏览器插入任何脚本，诸如获取用户 Cookie ，获取页面信息， 劫持页面跳转等，这些都能轻易办到。

浏览器的保护

浏览器当然是不会放任你的网站在 Internet 上裸奔的。我们可以在服务端设置响应头来让浏览器帮我们搞定XSS：

  • ctx.set('X-XSS-Protection', 0)
    这个响应头兼容性很不错。是Internet Explorer，Chrome和Safari的一个功能，当检测到跨站脚本攻击 (XSS)时，浏览器将停止加载页面。比如当我访问http://lizhangyu.com/id=<script>alert(1)</script>时，浏览器会禁止访问该网站，不过这么看起来，似乎X-XSS-Protection只能防御反射型XSS攻击，效果十分有限。
  • ctx.set('Content-Security-Policy', 'default-src \'self\'')
    这个响应头类似于跨域请求中的同源策略，只允许我们页面向我们允许的域名发起跨域请求,而不符合我们策略的恶意攻击则被挡在门外。更加深入了解请查阅内容安全策略（CSP） 。

CSRF-跨站请求伪造

CSRF（Cross-site request forgery）跨站请求伪造，也被称为“One Click Attack”（一点就爆炸？听起来有点吓人）或者Session Riding，通常缩写为CSRF或者XSRF，是一种对网站的恶意利用。尽管听起来像跨站脚本（XSS），但它与XSS非常不同，XSS利用站点内的信任用户，而CSRF则通过伪装来自受信任用户的请求来利用受信任的网站。与XSS攻击相比，CSRF攻击往往不大流行（因此对其进行防范的资源也相当稀少）和难以防范，所以被认为比XSS更具危险性。

UTF-8和讨厌的乱码

发表于 2018-08-15

想起小时候外盗版游戏的日子里，总会在打汉化补丁时出现一些不可名状的奇怪汉字构成的乱码，那时候只能靠连蒙带猜玩下去。想不到时过境迁，现在天天写代码的时候也是会经常遇到乱码。今天就来总结一下这讨厌的乱码究竟是什么在作怪吧！

汉字编码的痛楚

世界上存在着多种编码方式，当我们翻开大一C语言教材，里写的ASCII码就是最原始的编码了。ASCII 码一共规定了128个字符的编码，比如空格SPACE是32（二进制00100000），大写的字母A是65（二进制01000001）。这128个符号（包括32个不能打印出来的控制符号），只占用了一个字节的后面7位，最前面的一位统一规定为0。可是因为不同语言的问题，仅仅采用ASCII码显然行不通，于是便有了不同的编码规范，随之而来的，也就产生了乱码问题。
可以想象，如果有一种编码，将世界上所有的符号都纳入其中。每一个符号都给予一个独一无二的编码，那么乱码问题就会消失。这就是 Unicode，但是Unicode 只是一个符号集，它只规定了符号的二进制代码，却没有规定这个二进制代码应该如何存储。 而且，每一个unicode码都占用4个字节，按照二八法则，出现频率最高的数字和字母都采用4个字节的话将会极大的浪费空间。
随着互联网的发展，我们有了解决方法：UTF-8。
UTF-8 最大的一个特点，就是它是一种变长的编码方式。它可以使用1~4个字节表示一个符号，根据不同的符号而变化字节长度。
UTF-8 的编码规则很简单，只有二条：
1）对于单字节的符号，字节的第一位设为0，后面7位为这个符号的 Unicode 码。因此对于英语字母，UTF-8 编码和 ASCII 码是相同的。

2）对于n字节的符号（n > 1），第一个字节的前n位都设为1，第n + 1位设为0，后面字节的前两位一律设为10。剩下的没有提及的二进制位，全部为这个符号的 Unicode 码。
好了，了解到了UTF-8，我们再来看看中文编码。中国大陆最常用的就是GBK18030编码，除此之外还有GBK，GB2312，这几个编码的关系是这样的。 最早制定的汉字编码是GB2312，包括6763个汉字和682个其它符号 95年重新修订了编码，命名GBK1.0，共收录了21886个符号。 之后又推出了GBK18030编码，共收录了27484个汉字，同时还收录了藏文、蒙文、维吾尔文等主要的少数民族文字。而这些编码的特点是，他们是定长编码。这就是我们出现中文乱码的原因，本质就是：读取二进制的时候采用的编码和最初将字符转换成二进制时的编码不一致。 乱码的原因一般是编码和解码的方式不是用的同一套格式所致，要想恢复乱码，必须对症下药，找到当初的编码格式，这样一般都能解决乱码问题。

MySQL 中使用「utf8」

好吧，既然编码规范这么烦杂，我们干脆都用UTF-8吧。理论上来讲这当然是个好办法，但是因为历史遗留问题，我们的数据库可就没那么听话。
事实上，MySQL使用的「utf8」只支持每个字符最多三个字节，而真正的 UTF-8 是每个字符最多四个字节。MySQL 一直没有修复这个 bug，他们在 2010 年发布了一个叫作「utf8mb4」的字符集，绕过了这个问题。当然，他们并没有对新的字符集广而告之（可能是因为这个 bug 让他们觉得很尴尬），以致于现在网络上仍然在建议开发者使用「utf8」，但这些建议都是错误的。为什么会造成这种局面呢，可能的原因是MySQL诞生的时候UTF-8标准还没有得到广泛推广。

所以，当你使用数据库时，请使用utf8mb4。

flow那些事儿

按照flow官网上面的解释，Flow 是 facebook 出品的 JavaScript 静态类型检查工具。js界流传着一句话：动态类型一时爽，代码重构火葬场。 google的angular团队作为一群java出身的程序员，当然受不了javascript这糟糕的特性。于是选择了typescript的作为开发语言。而angular从诞生之初就带着浓厚的java面向对象的特点，这也使得前端程序员的上手门槛变高。react和后来的vue选择了更为前端接受的flow。

三大框架的解决方案：
|angular | react | vue |
|—|—|—|
| typescript | flow | flow |

flow的类型检查分为以下两种方式：

  • 通过注释：类似于typescript，事先注释好我们期待的类型，Flow就会基于这些注释来评估。
  • 通过代码推断：通过变量的使用上下文来推断出变量类型，然后根据这些推断来检查类型。

第一种方式，我们需要额外编写只在开发阶段起作用的代码，最后在代码编译打包的阶段被剔除。显然，这种额外添加类型注释的方式增加了工作量。

第二种方式，不需要任何代码修改即可进行类型检查，最小化开发者的工作量。它不会强制你改变开发习惯，因为它会自动推断出变量的类型。这就是所谓的类型推断，Flow最重要的特性之一。

至于为什么选用flow，尤雨溪在知乎上做过解释：
Vue 2.0 本身在初期的快速迭代阶段是用 ES2015 写的，整个构建工具链也沿用了 Vue 1.x 的基于 ES 生态的一套（Babel, ESLint, Webpack, Rollup…)，全部换 TS 成本过高，短期内并不现实。相比之下 Flow 对于已有的 ES2015 代码的迁入/迁出成本都非常低。

在 Vue.js 的主目录下有.flowconfig 文件， 它是 Flow 的配置文件，这其中的 [libs] 部分用来描述包含指定库定义的目录，默认是名为 flow-typed 的目录。
这里 [libs] 配置的是 flow，表示指定的库定义都在 flow 文件夹内。我们打开这个目录，会发现文件如下：
flow
├── compiler.js // 编译相关
├── component.js // 组件数据结构
├── global-api.js // Global API 结构
├── modules.js // 第三方库定义
├── options.js // 选项相关
├── ssr.js // 服务端渲染相关
├── vnode.js // 虚拟 node 相关
可以看到，Vue.js 有很多自定义类型的定义，在阅读源码的时候，如果遇到某个类型并想了解它完整的数据结构的时候，可以回来翻阅这些数据结构的定义。

vue的目录结构

src
├── compiler // 编译相关
├── core // 核心代码
├── platforms // 不同平台的支持
├── server // 服务端渲染

vue的项目结构是由rollup完成的，比起webpack，rollup更加轻量。只负责对js文件进行打包。

compiler

目录包含 Vue.js 所有编译相关的代码。它包括把模板解析成 ast 语法树，ast 语法树优化，代码生成等功能。
编译的工作可以在构建时做（借助 webpack、vue-loader 等辅助插件）；也可以在运行时做，使用包含构建功能的 Vue.js。显然，编译是一项耗性能的工作，所以更推荐前者——离线编译。

core

core 目录包含了 Vue.js 的核心代码，包括内置组件、全局 API 封装，Vue 实例化、观察者、虚拟 DOM、工具函数等等。
这里的代码可谓是 Vue.js 的灵魂，也是我们之后需要重点分析的地方。

platform

Vue.js 是一个跨平台的 MVVM 框架，它可以跑在 web 上，也可以配合 weex 跑在 natvie 客户端上。
Vue.js 2.0 支持了服务端渲染，所有服务端渲染相关的逻辑都在这个目录下。
服务端渲染主要的工作是把组件渲染为服务器端的 HTML 字符串，将它们直接发送到浏览器，最后将静态标记”混合”为客户端上完全交互的应用程序。

后记：浅析Hexo原理
Hexo这么好用，那么它的实现原理是怎么样的呢？
首先我们知道Github Pages支持静态页面的解析，也就是说我们可以直接把html页面push到Github Pages上。Hexo所做的就是把你写的markdown文件和主题模板结合，生成html静态文件，最终由Github Pages展现出来。
这里我们来分析一下 Hexo 每次部署的流程：
hexo generator：生成静态文件。遍历主题文件中的 source 文件夹（js、css、img 等静态资源），然后建立索引，然后根据索引生成 pubilc文件夹中，此时的 publid 文件是由 html、 js、css、img 建立的纯静态文件可以通过index.html作为入口访问你的博客。
hexo deploy：部署文件。部署主要是根据在 _config.yml 中配置的 git 仓库，将 public 文件上传至 github 中。然后再根据上面的 github 提供的 pages 服务呈现出页面。当然你也可以直接将你生成的 public 文件上传至你自己的服务器上。
可以看到，从markdown到html的过程由generator指令完成，这个过程进行了两次模板渲染：
第一次渲染：将sourse目录下的文件渲染为一个article对象。

其中模块hexo-renderer-marked 用于解析md文件，js-yaml 是解析yml文件。

第二次渲染：将article对象和themes文件通过模板引擎（Hexo默认为ejs）渲染为静态页面资源，保存在public文件夹中。
